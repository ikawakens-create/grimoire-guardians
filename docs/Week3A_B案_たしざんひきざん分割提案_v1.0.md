## 📋 経緯と目的

### 提案の背景

Week 3A の実装計画（Gemini承認済み）では、Phase 0.1 のワールド構成は以下です:

```
world_1: M1-01  なかまづくりと かず    15問
world_2: M1-02  10までの かず         18問（筆記3問はダミー）
world_3: M1-03  なんばんめ            16問
world_4: M1-04  いくつと いくつ        25問

world_6: M1-06  のこりは いくつ        20問  ← ひきざん導入

合計: 6ワールド・114問
```

**人間開発者からの提案:**
> 「小学1年生の足し算と引き算は1番重要な項目なので、
> 問題数を増やすもしくは足し算や引き算が終わったあとに
> 足し算2・引き算2のワールドを追加するなどを提案します」

これを受けClaudeが3案を提示。**B案が有力候補**として選ばれたため、
Geminiに最終的な判断を依頼する。

## 5. A案との詳細比較

### 5-1. 子どもの体験比較

```yaml
A案（増量: 30問）の子どもの体験:
  「たしざん の ほんを えらんだ」
  → 問題1〜10: 楽しい！
  → 問題11〜20: まだある…
  → 問題21〜30: 長い…早く終わりたい
  → クリア後: 「やっと おわった」感

B案（分割: 15問 + 18問）の子どもの体験:
  「たしざん きほん を えらんだ」
  → 問題1〜15: 楽しい！（ちょうどいい長さ）
  → クリア: 「🎉 きほん クリア！！」（達成感）
  → 「おうよう に チャレンジ！」
  → 問題1〜18: 文章題が面白い、頭を使う感覚
  → クリア: 「🎉🎉 おうよう も クリア！！」（大きな達成感）
```

### 5-2. 教育観点の比較

```yaml
A案:
  ✅ 連続した学習体験
  ⚠️ 難易度の段差なし（全問同じ難度になりがち）
  ⚠️ 長すぎると集中力が切れる（特に6〜7歳）

B案:
  ✅ きほん → おうようの自然な難易度段差
  ✅ 文章題リーディング力も同時に育てられる
  ✅ 「式を選ぶ」「□を求める」で多様な思考を刺激
  ✅ 15〜18問は1年生の集中力に適切
  ⚠️ おうようが難しすぎると挫折感になる（→ 難易度調整が重要）
```

## 6. 懸念事項

### 6-1. 難易度設計

```yaml
懸念: 「おうよう」が難しすぎると挫折する

対策案:
  1. きほんをクリアしてから解放される（準備ができた状態で挑む）
  2. ヒントボタンを「きほん」より多く配置
  3. おうようのCLEAR_THRESHOLDを下げる案
     きほん: 60%（通常通り）
     おうよう: 50%（文章題は難しいので少し易しめ）
     → Gemini Q4 で判断を仰ぐ

Geminiへ: おうようワールドのCLEAR_THRESHOLDを下げるべきか？
```

### 6-2. 本棚の見た目

```yaml
懸念: 8ワールドになると本棚画面が横スクロールや
      2行になる可能性がある

現状のBookshelfScreen:
  - 最大6冊を想定した横スクロール実装
  - 8冊になった場合のUIは未設計

対策案:
  A: 横スクロールをそのまま延ばす（8冊対応）
  B: 2×4グリッドにする
  C: 「Phase 0.1」と「Phase 0.2」でグループ分けして表示
  D: 実装コストを抑えるため、A案（各ワールド増量）に変更

Geminiへ: UI的にどの対策が最善か？
```

### 6-3. world_5b 強制かスキップ可か

```yaml
懸念: world_5b（おうよう）はクリア必須か否か

オプション1: 必須（world_5b をクリアしないと world_6 に進めない）
  → 全員が応用力を身につける
  → 詰まった子が先に進めない

オプション2: 任意スキップ可（world_5 → world_6 の直接ルートあり）
  → 早く先に進みたい子はスキップ
  → コンテンツが無駄になるリスク

Geminiへ: 必須 or スキップ可、どちらが教育的に正しいか？
```

### 6-4. セーブデータ互換性

```yaml
懸念: B案を採用する場合、_getInitialState() に
      world_5b / world_6b を追加する必要がある

Week 3A でSaveManagerを実装する（Gemini承認済み）ので、
今のうちに初期状態に含めておけば問題ない。

ただし、仮に Week 3A を B案なしで実装した後に B案を追加する場合、
セーブデータのマイグレーションが必要になる。

推奨: Week 3A 開始前に B案の採否を決定する
```
